### 产品

#### 1. x.p1gd0g.cc

遇到了不少问题，包括一些技术选择。按顺序写一下。

1. 先是写了一个请求的缓存机制，直接把结果存在 prefs。
2. 尝试前端加入 sql 缓存，未果。drift 需要开启 wasm，且引用过程有点复杂。换思路 sql 加在后端。
3. 后端有三个思路：接在 pocketbase 上，接在 cf sql 上，接在自己的后台上。
4. 在这个问题上纠结了蛮久。首先排除第三个，要花过多的人力了。接着看了下 cf sql 的 limit，处在一个尴尬的位置上，database 多了也不行，单个库 size 大了也不行。
5. 最后接在 pocketbase 上，算是最简单的方案了。
6. 调试过程中发现 web prefs 只有 5mb，dllm。只能再把请求缓存去掉。除非换成高级的 nosql 数据库。
7. 想了下如果有了 sql 缓存，也不太需要请求缓存了，先不接 nosql。

目前 sql 缓存 put 过程已发布，接下来就是 get 部分。get 部分需要的假期判断已经写好。没有使用 api 而是直接把 json 文件放在 pocketbase 上，每年更新一次就好，维护还算方便。野生的 api 实在不太敢用。

还有就是几个外链有用户反馈无法访问，被当成是软件本身的问题，头疼，这么久第一次收到差评。直接复制粘贴了一份到公众号。
